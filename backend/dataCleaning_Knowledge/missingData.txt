Here's the full breakdown of every decision made.
GET /task2/missing_info — the analysis endpoint.
It loops through every column and skips any that have zero missing values — no point showing the user a clean column. For each column that does have missing data, it calculates the count and percentage, assigns a severity band (low/medium/high), and then builds a strategies dict. Each strategy entry has a label for the dropdown and an explanation from xai/explainer.py that tells the user exactly what will happen if they pick that option — before they click Apply.
Mean and median are only offered for numeric columns. Mode is offered for every column because it works on both numbers and text. The frontend gets everything it needs to render the full interactive card in one single GET call.
POST /task2/fix_missing — the write endpoint.
The key design decisions here: dropna(subset=[col]) is used instead of dropna() because the plain dropna() drops any row missing a value in any column — which is almost never what you want. subset=[col] only drops rows where that specific column is missing. After dropping rows, reset_index(drop=True) renumbers the index from 0 again — without this, you get gaps in the index which causes confusing behaviour later. fillna(inplace=True) modifies the column directly inside the copy, and as always the copy is only written back to dataset_state.df at the very end if at least one change succeeded.
xai/explainer.py now has Task 2 appended at the bottom. Every explanation is a separate function that takes the real column data as arguments — so the message says "missing 3.2% of its values" not just a generic sentence. The severity function has three completely different responses depending on whether the missing rate is low, medium, or high.
