Here's the full breakdown of every decision.
Three endpoints instead of two. Tasks 1–5 each had one read endpoint and one write endpoint. Task 6 has three because validation has a natural three-step flow — first you see the suggestions (GET /validation_info), then you run the checks (POST /run_validation), then you fix what failed (POST /fix_violations). Separating run and fix is important because the user needs to review violations before deciding how to handle them. Running and fixing in one call would skip that review step entirely.
detect_auto_rules() — the proactive xAI layer. Instead of making the user know what rules to apply, the system scans every column name and infers sensible rules. A column containing "year" automatically gets a no_future_year suggestion. A column containing "pct" gets a percentage rule. This is domain knowledge encoded as logic — the user benefits from it without needing to know it exists.
run_single_rule() is a pure function. It takes a DataFrame, column, rule, and params, and returns a result dict. It never modifies anything. This design means both run_validation and fix_violations can call it — run_validation uses it to report, and fix_violations re-runs it to get the current violation mask before applying the fix. Re-running is safer than storing indices from a previous call because the DataFrame may have changed between the two requests.
df.loc[violation_indices, col] is the cleanest way to update specific rows. We target only the rows that violated the rule by their index, so no other rows are touched. This is why run_single_rule returns violation_indices in its result — so fix_violations can use them directly.
The set_null fix method is deliberately honest. When a value is wrong but the correct value is unknown, setting it to NaN is more truthful than guessing a replacement. The user can then go back to Task 2 to handle those new missing values with the right strategy for their data.
reset_index(drop=True) after drop_rows. Same as every other task — dropping rows leaves gaps in the index that cause subtle bugs in downstream operations. Always reset after any row removal.
