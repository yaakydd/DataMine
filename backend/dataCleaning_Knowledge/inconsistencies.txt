The file has three layers.
The two helper functions at the top — require_df() and auto_clean_name() — are pure utility. They don't touch any endpoint logic. detect_issues() sits just below them and is the xAI brain of this task. It doesn't fix anything, it just looks, thinks, and explains.
auto_clean_name() — why the order of the 6 rules matters.
Rules 1 and 2 must come before Rule 3. If you ran Rule 3 first — replacing all non-word characters — spaces would become underscores, but so would nothing else in a useful way. By handling spaces explicitly in Rule 2 first, the intent is clear. Rule 4 must come after Rule 3 because Rule 3 is what creates multiple consecutive underscores. Rule 5 must come after Rule 4 because collapsing ___ into _ might leave you with a leading or trailing single _. Rule 6 is always last because none of the earlier rules care about casing.
detect_issues() — the xAI layer.
Check 4 is the most interesting. A column called "age" with dtype object looks fine by name, but if its values are "25", "31", "28", it's actually a numeric column disguised as text. The trick is stripping $, %, and commas before calling float() — otherwise "$4,500" would fail the float test and get missed.
update_columns() — why .copy() is the most important line.
If you work directly on dataset_state.df, a failed dtype conversion halfway through leaves your data in a half-changed, unrecoverable state. .copy() means the original is always safe. dataset_state.df = df only runs at the very end, and only if something actually changed.

GET /task1/infer_types — Data type inference.
Three checks run on every column. The hidden boolean check uses series.unique() and tests whether the set of values is a subset of {0, 1} — if a float column only ever contains 0.0 and 1.0, it's almost certainly a flag that should be bool. The ID check combines two signals — all values unique AND the column name contains a known identifier keyword — because uniqueness alone isn't enough (a name column might also be unique). The high cardinality check only fires when uniqueness exceeds 50% AND there are more than 10 unique values, avoiding false positives on tiny datasets.
GET /task1/text_value_info — Text value scanning.
The whitespace detection uses series != series.str.strip() — a vectorised comparison that's far faster than looping. The casing detection compares series.nunique() (raw) against series.str.lower().nunique() (normalised) — if lowercasing reduces the unique count, casing variants exist. The groupby(series.str.lower()) then reconstructs the exact variant groups so the user sees {"male": ["Male", "male", "MALE", "M "]} not just a count.
POST /task1/clean_text_values — Text value cleaning.
The original_nulls = df[col].isna() line before every operation and df.loc[original_nulls, col] = np.nan after is the critical safety step. Pandas .str operations silently convert NaN to the string "nan" in some versions. Saving the null positions first and restoring them after prevents NaN values from becoming the literal string "nan" in your dataset.
POST /task1/harmonise_categories — Category harmonisation.
The mapped.combine_first(df[col]) pattern is the key design decision. .map() returns NaN for any value not found in the mapping dict — which would erase all unmapped values. combine_first() fills those NaN positions with the original values, meaning only the explicitly mapped variants get replaced and everything else stays exactly as it was.
