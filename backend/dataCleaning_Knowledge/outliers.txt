Why both Z-score and IQR. They're not redundant — they disagree on borderline cases in meaningful ways. Z-score uses the mean and standard deviation as its reference points, but those are both already pulled by outliers. If your salary column has three people earning $10M in a dataset of $50K earners, the mean shifts upward and the std inflates, which actually makes the extreme values look less extreme by Z-score. IQR is built on percentiles which are completely immune to that problem. Showing both gives the user a fuller picture and lets them cross-reference.
std == 0 guard. If every value in a column is identical — say, a "category" column accidentally typed as numeric with all values being 1 — the standard deviation is zero and the Z-score formula divides by zero. The guard catches this before numpy throws a ZeroDivisionError.
series.dropna() before all calculations. NaN propagates through every numpy operation silently — np.mean([1, 2, np.nan]) returns nan, not 1.5. Dropping NaN before computing statistics ensures clean results. We're not removing them from the DataFrame — just from the calculation.
clip() for capping. This is the cleanest way to cap values in pandas. df[col].clip(lower=lower_fence, upper=upper_fence) replaces every value below the lower fence with the lower fence value, and every value above the upper fence with the upper fence value, in one vectorised operation. No loops, no conditionals.
within_fences | is_null when removing rows. When we drop outlier rows, we use (value >= lower) & (value <= upper) | is_null. The is_null part is critical — without it, any row that has a NaN in this column would also get dropped, which is wrong. NaN handling is Task 2's job, not Task 4's. Each task stays in its own lane.
The "do_nothing" save guard. At the end, we check not all("left as-is" in a for a in applied) before saving back. If the user sent a strategy of do_nothing for every column, there's nothing to write back to dataset_state.df. This avoids an unnecessary copy operation.

Why Mahalanobis and not Isolation Forest. Isolation Forest is a tree-based ML algorithm that requires fitting a model. Mahalanobis distance is pure statistics — it only needs a mean vector and a covariance matrix, both computed directly from the data. No training, no hyperparameters to tune, no model to worry about. It belongs firmly in data analysis.
The singular matrix guard. When two columns are perfectly correlated — like total_price and unit_price * quantity — the covariance matrix cannot be inverted because it has no unique solution. np.linalg.inv() raises LinAlgError in this case. We catch it and return explain_mahalanobis_singular_matrix() which tells the user exactly what went wrong and points them to the correlation analysis to fix it. Without this guard the endpoint crashes with a cryptic numpy error.
max(dist_sq, 0.0) before np.sqrt(). Floating point arithmetic on near-singular matrices can produce tiny negative numbers like -1e-15 for the squared distance. np.sqrt() of a negative number returns nan. The max(..., 0.0) clamps it to zero so the distance is always a real number.
Chi-squared threshold. Squared Mahalanobis distances follow a chi-squared distribution with degrees of freedom equal to the number of columns used. stats.chi2.ppf(0.975, df=n_cols) gives the value below which 97.5% of distances fall — anything above it is in the extreme 2.5% and gets flagged. This is a principled statistical threshold, not an arbitrary number.
flag vs drop in the fix endpoint. Dropping is permanent and irreversible. Flagging adds a __multivariate_outlier__ boolean column so the user can keep the rows in the dataset while clearly marking them for separate analysis. This is often the right choice when you are not certain whether the combination is an error or a genuine extreme.
Upper triangle only in correlation. corr_matrix.loc[col1, col2] and corr_matrix.loc[col2, col1] are identical — the matrix is symmetric. Using i < j means each pair appears exactly once in the results. Without this you would see every pair twice and the suggested_drop logic would contradict itself.

Here is exactly why the previous .item() approach on .loc didn't work and why this approach does.
corr_matrix.loc[col, other] is typed by pandas stubs as returning Scalar, which is the union str | bytes | date | datetime | bool | int | float | complex | Timestamp | Timedelta. Most of those types don't have .item() defined — only numpy scalars do. So Pylance correctly flagged .item() as potentially calling a non-existent method on str, bool, int, datetime, etc.
The fix is to never call .loc at all in the correlation section. Instead:
pythoncorr_array = corr_df.to_numpy(dtype=float)
.to_numpy(dtype=float) converts the entire DataFrame into a plain np.ndarray with dtype=float64. Every element is now np.float64 — a numpy scalar with .item() always defined. Indexing with corr_array[i, j] gives np.float64, .item() on that gives a plain Python float, and round(float, 4) and abs(float) and passing to explain_high_correlation_pair(..., corr_val: float) all work perfectly with no complaints from Pylance.
